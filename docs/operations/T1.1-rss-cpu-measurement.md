# T1.1 — RSS/CPU-Messung: `tsx` vs. `node dist/` (V5 Spur 1)

**Zweck:** Den DoD-Teil „RSS/CPU vorher/nachher gemessen" der T1.1-Startumstellung
(`tsx` → `node dist/`) reproduzierbar erheben. Die **Code-Umstellung ist bereits gemergt**
(PR #217: `start`/`daemon:start` → `node dist/index.js`, `start:tsx` als Dev-Fallback,
systemd `ExecStart` → `node`). Offen ist nur der **Zahlen-Nachweis** — der braucht einen
Live-Lauf und ist daher hier als Runbook beschrieben (kein Teil eines Code-PRs).

## Vorbereitung

```bash
# Build muss existieren (node dist/ startet nur mit kompiliertem Code):
npm run daemon:build          # cd packages/daemon && npx tsc
ls packages/daemon/dist/index.js
```

Messumgebung fixieren, damit vorher/nachher vergleichbar ist:
- **Gleicher Host, gleiche Env** (`TLMCP_*`), gleicher `TLMCP_DATA_DIR`, gleiches Netz.
- **Keine anderen Lastquellen** auf dem Host während der Messung.
- Zwei Phasen je Modus: **idle** (nur Heartbeat) und **unter Last** (optional, z.B.
  `discover_peers`/Inbox-Polling aktiv), getrennt messen.

## Zwei Läufe

> **Wichtig:** `$!` liefert die **npm-Wrapper**-PID, nicht den Daemon-Prozess (bei
> `daemon:start` läuft zusätzlich `daemon:build` davor). Immer die **echte** Daemon-PID
> per `pgrep` bestimmen, sonst misst man ~0.

**Vorher (tsx):**
```bash
npm run start:tsx &
sleep 60                       # Warmup
DAEMON_PID=$(pgrep -f 'tsx src/index.ts')
```

**Nachher (node dist/):**
```bash
npm run daemon:start &
sleep 60                       # Warmup (inkl. daemon:build)
DAEMON_PID=$(pgrep -f 'dist/index.js')
```

> Jeweils erst ~60 s **Warmup** abwarten (Boot, mDNS, libp2p, erste Heartbeats),
> dann messen.

## Sampeln

Der Sampler misst den **gesamten Prozessbaum** (root + alle Nachfahren): pro Tick
`ps -e -o pid=,ppid=` → Baum von `<pid>` bestimmen → `ps -o rss=,%cpu= -p <alle>` →
**Σ RSS / Σ CPU**. Das ist entscheidend für Fairness: der `tsx`-Start ist ein Baum
(node + esbuild-Transform-Kind), `node dist/` ein Einzelprozess — nur die Baum-Summe
ist vergleichbar. `<pid>` = die **root**-Daemon-PID (aus `pgrep`, s.o.). Verdichtung zu
`mean/p50/p95/min/max` über den reinen Helfer `rss-cpu-stats.ts`:

```bash
# Standard: 60 Samples, 1 s Intervall (= 60 s Fenster). Ausgabe = JSON-Summary.
node scripts/measure-daemon-rss-cpu.mjs --pid "$DAEMON_PID" --samples 60 --interval-ms 1000 > before-tsx.json
# … zweiter Lauf …
node scripts/measure-daemon-rss-cpu.mjs --pid "$DAEMON_PID" --samples 60 --interval-ms 1000 > after-node.json
```

## Vergleich

```bash
node scripts/measure-daemon-rss-cpu.mjs --compare before-tsx.json after-node.json
```

erzeugt die Markdown-Vergleichstabelle (`formatComparison`), z.B.:

```text
| Metrik | tsx (vorher) | node dist (nachher) | Δ |
|---|---|---|---|
| RSS mean (MiB) | … | … | …% |
| RSS p95 (MiB)  | … | … | …% |
| CPU mean (%)   | … | … | …% |
```

## Akzeptanz / Ablage

- Beide Phasen (idle, ggf. Last) je Modus mit **n ≥ 60** Samples.
- Erwartung: `node dist/` ohne den tsx-Loader ⇒ **niedrigeres RSS**, gleiche oder
  geringere CPU-Grundlast (kein `esbuild`-Transform zur Laufzeit). Reale Zahlen entscheiden.
- Ergebnis-Tabelle in den T1.1-Abschluss (CHANGES/Report) übernehmen; Roh-JSONs beim
  Ausführenden ablegen. **Kein Zahlen-Erfinden** — nur gemessene Werte.

## Ergebnis — Live-Messung 2026-07-02 (TH01/ThinkHub)

Erhoben mit exakt dem Runbook oben; beide Läufe **identisch konfiguriert**, gleicher
Host, gleicher `TLMCP_DATA_DIR` (dieselbe Identität/Cert), je **n = 60** Samples @ 1 s,
20 s Warmup. Prozessbaum-Sampling (root + Nachfahren) via `measure-daemon-rss-cpu.mjs`
(tsx = node-tsx-Runtime + Daemon-Kind; node dist = Einzelprozess).

| Metrik | tsx (vorher) | node dist (nachher) | Δ |
|---|---|---|---|
| RSS mean (MiB) | 215.8 | 129.1 | **-40.2%** |
| RSS p95 (MiB)  | 216.6 | 129.8 | **-40.1%** |
| RSS max (MiB)  | 216.6 | 129.8 | **-40.1%** |
| CPU mean (%)   | 4.82  | 2.63  | **-45.5%** |
| CPU p95 (%)    | 9.00  | 4.90  | **-45.6%** |

_n(vorher)=60, n(nachher)=60 Samples._

**Verdikt:** Die Startumstellung `tsx` → `node dist/` senkt den RSS-Fußabdruck um
**~40 %** (≈ 87 MiB) und die CPU-Grundlast um **~46 %**. Ursache: kein `tsx`/esbuild-
Transform-Prozess und keine In-Memory-Source-Transformation zur Laufzeit. Die Erwartung
des Runbooks (niedrigeres RSS, gleiche/geringere CPU-Grundlast) ist **empirisch bestätigt**.

**Isolations-Caveat (Ehrlichkeit zur Umgebung):** Gemessen auf einer **isolierten**
Instanz — `TLMCP_RUNTIME_MODE=local` (loopback-bind), `TLMCP_LIBP2P_ENABLED=0`,
`TLMCP_MDNS_ENABLED=0`, Port 9460, temporärer data dir — um den laufenden Produktiv-
Daemon (Port 9440) und den LAN-Mesh **nicht** zu stören. Die **Absolutwerte** liegen
daher unter einem voll vermaschten Produktions-Daemon; das **Δ (tsx→node)** ist bei
identischer Konfiguration beider Läufe das belastbare DoD-Signal — der tsx-Loader-
Overhead ist weitgehend last-unabhängig additiv. Roh-JSONs eingebettet in
`changes/2026-07-02_t11-rss-cpu-live-measurement.md`.
